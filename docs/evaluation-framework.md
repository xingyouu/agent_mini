# Mini Agent 完整度评估方案

## 1. 评估目标

本方案用于评估一个 Mini Agent 项目是否具备可持续扩展的工程基础。它不只检查“能不能跑起来”，还关注核心闭环、工具能力、上下文治理、长期记忆、技能路由、主动调度、网关服务和异常恢复。

评估结果分为 10 个维度，总分 100 分。每个维度既可以人工评分，也可以逐步沉淀为自动化回归用例。

## 2. 当前结构盘点

| 观察项 | 证据 | 结论 |
| --- | --- | --- |
| Agent 主体 | `src/agent.ts` | 已具备统一入口、事件分发和运行编排 |
| Agent Loop | `src/agent-loop.ts` | 已具备 follow-up、steering 和 tool loop 机制 |
| 会话持久化 | `src/session.ts` | 已具备 JSONL 会话保存和读取能力 |
| 上下文治理 | `src/context/*` | 已具备加载、裁剪、压缩和窗口保护 |
| 内置工具 | `src/tools/builtin.ts` | 已具备文件、命令、记忆等基础工具 |
| 长期记忆 | `src/memory.ts` | 已具备保存、检索和读取能力 |
| 技能系统 | `src/skills.ts` | 已具备 `SKILL.md` 加载、合并和匹配能力 |
| 主动调度 | `src/heartbeat.ts` | 已具备 interval、requested 等触发机制 |
| Gateway | `src/gateway/server.ts` | 已具备 HTTP + WebSocket 服务 |
| RPC 方法 | `src/gateway/handlers.ts` | 已具备连接、聊天、历史、会话和健康检查方法 |
| Web UI | `src/gateway/web-ui.ts` | 已具备浏览器侧操作界面 |
| 工程保护 | `src/tool-approval.ts`、`src/tool-policy.ts`、`src/context-window-guard.ts`、`src/session-tool-result-guard.ts` | 已覆盖审批、策略、上下文保护和工具结果修复 |

初步判断：项目已经超过普通 Agent Demo，属于“骨架较完整、工具链较全、网关能力可用”的阶段。下一步重点应放在自动化测试、端到端验证和真实任务质量评估。

## 3. 评分模型

| 维度 | 分值 | 核心问题 |
| --- | ---: | --- |
| A. 核心闭环完整度 | 10 | 是否形成输入、推理、工具、结果输出的完整闭环 |
| B. 工具使用能力 | 10 | 是否能正确选择工具、调用工具并解释结果 |
| C. 会话与上下文管理 | 10 | 是否能跨轮保持状态，并在上下文受限时保留主线 |
| D. 长期记忆能力 | 10 | 是否能存、找、取、用长期信息 |
| E. 技能与任务路由 | 10 | 是否能识别何时使用技能，并按技能说明执行 |
| F. 主动性与调度 | 10 | Heartbeat 是否稳定触发并产出合理动作 |
| G. Gateway 与多端可用性 | 10 | Gateway、Web UI、Client 是否行为一致且可恢复 |
| H. 安全与约束 | 10 | 审批、沙箱、工具策略是否生效 |
| I. 健壮性与异常恢复 | 10 | 工具失败、上下文溢出、断连后是否可恢复 |
| J. 工程可扩展性 | 10 | 模块边界、配置能力、二次开发成本是否合理 |

## 4. 评分口径

| 分值 | 判定标准 |
| ---: | --- |
| 0 | 功能不存在，或完全不可用 |
| 3 | 功能存在，但大多数场景不可用 |
| 5 | 基本可用，但不稳定或覆盖不足 |
| 8 | 大部分场景可用，只剩边界问题 |
| 10 | 功能稳定、覆盖全面、边界处理完整 |

## 5. 基线建议分

| 维度 | 基线分 | 说明 |
| --- | ---: | --- |
| A. 核心闭环完整度 | 9 | 主循环、事件流、工具回路、会话写入均已存在 |
| B. 工具使用能力 | 8 | 内置工具覆盖较广，但复杂工具生态仍有限 |
| C. 会话与上下文管理 | 8 | 已有 pruning、compaction 和 context guard |
| D. 长期记忆能力 | 6 | 机制完整，但当前检索策略较轻量 |
| E. 技能与任务路由 | 7 | 技能匹配机制已具备，需要更多真实任务验证 |
| F. 主动性与调度 | 6 | Heartbeat 已实现，主动效果质量仍需评估 |
| G. Gateway 与多端可用性 | 8 | Gateway、Client、Web UI 已存在，需做多端稳定性测试 |
| H. 安全与约束 | 8 | 审批、策略、沙箱保护设计较完整 |
| I. 健壮性与异常恢复 | 7 | 已有工具结果修复、上下文保护和重试思路 |
| J. 工程可扩展性 | 8 | 模块分层清晰，扩展点较多 |

基线总分：`75/100`。

## 8. 阶段判定

| 总分 | 判定 |
| ---: | --- |
| 0-39 | 仅有概念验证，完整度低 |
| 40-59 | 基础可用，但模块协同弱 |
| 60-74 | 架构基本完整，但缺少验证闭环 |
| 75-89 | 完整度较高，具备演示与扩展价值 |
| 90-100 | 接近生产级验证完成 |

## 9. 下一步建议

1. 将 30 条评估用例整理为可重复运行的数据文件。
2. 增加 CLI、Gateway、Memory 的端到端自动化回归。
3. 为每个维度补充 pass/fail 示例，降低人工评分偏差。
4. 引入成功率、平均耗时、平均工具调用数和异常恢复率等稳定性指标。
